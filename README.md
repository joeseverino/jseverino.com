# jseverino.com

Astro rebuild of `jseverino.com`.

This repo is the public website build target. The editable source content lives
in the private Severino Labs vault, then gets synced into this repo as a public
snapshot that Cloudflare Pages can build without vault access.

## How It Works

```text
Severino Labs vault       private source of truth
  06 Pages/               editable site pages and site settings
  05 Writeups/            editable portfolio writeups and screenshots

sync-content              copies only public content/assets into this repo

jseverino-site            Astro code, synced public content, public assets

Cloudflare Pages          builds this repo and serves dist/
```

The split is intentional:

- The vault stays private and can contain drafts, notes, and internal metadata.
- This repo contains only the code and public content required to build the site.
- Cloudflare Pages does not need access to the vault.

## What To Edit

Edit content in the vault, not directly in this repo:

- Site navigation: `Severino Labs/06 Pages/_site.md`
- Technology groups: `Severino Labs/06 Pages/_technology-groups.md`
- Pages: `Severino Labs/06 Pages/<page>/index.md`
- Page images: `Severino Labs/06 Pages/<page>/images/`
- Portfolio writeups: `Severino Labs/05 Writeups/<slug>/index.md`
- Writeup images: `Severino Labs/05 Writeups/<slug>/images/`

Files under `src/content/` and `public/assets/pages|writeups/` are generated by
the sync step. They are committed so Cloudflare can build this repo by itself,
but they should not be hand-edited.

## Local Workflow

From anywhere, using the personal tools suite:

```sh
site status
site sync
site check
site publish
```

From inside this repo, the equivalent npm workflow is:

```sh
npm run sync:content
npm run check
npm run publish:check
```

`site publish` / `npm run publish:check` runs the full pre-push path:

1. Remove generated build output.
2. Sync public pages and writeups from the vault.
3. Run Astro diagnostics.
4. Build the static site.
5. Audit published image weight.

After that, commit and push this repo. Cloudflare Pages builds `dist/`.

## Content Rules

Pages and writeups publish only when their vault frontmatter uses the explicit
publish gate:

```yaml
published: true
```

The card and SEO summary belongs in frontmatter only:

```yaml
description: >-
  Short card and SEO summary.
cover_image: ./images/cover.png
technologies:
  - tailscale
  - docker
featured: true
featured_order: 1
```

Do not repeat the description as the first paragraph or blockquote. The sync
script strips exact repeated descriptions as a safety net.

## Page Blocks

Use normal Markdown for prose. These directives cover the custom page blocks the
site currently needs:

```md
::buttons
- [Primary Button](/resume/)
- [Secondary Button](/contact/)
::

::button
[Single Button](/portfolio/)
::

::button sticky
[Download Resume](/resume.pdf)
::

::center
Centered Markdown goes here.
::

::split
![Image](./images/example.jpg)
:::
Text for the right column.
::

::featured-projects
::

::technology-cloud
::

::contact-form
::
```

For captions, put a short paragraph immediately after the image:

```md
![Dashboard](./images/dashboard.png)

Dashboard after enabling alerts.
```

## Images

Keep source screenshots beside the markdown in `images/`. The sync script copies
only images and files that are referenced by markdown or frontmatter, so unused
large originals can stay in the vault without shipping to Cloudflare.

Use:

```sh
site publish
```

or:

```sh
npm run audit:assets
```

to catch oversized published images before pushing.

## Repo Boundaries

Commit:

- Astro source in `src/`
- public static files in `public/`
- synced public content in `src/content/`
- package and config files

Do not commit:

- `node_modules/`
- `.astro/`
- `dist/`
- `.env*`

The operational details live in the private vault:

- Infrastructure: `02 Infrastructure/Cloudflare Pages/Astro Static Site Stack.md`
- Runbook: `03 Runbooks/Publish a Writeup.md`
